# Vault Protocol Specification

## Overview

The Vault Protocol defines the universal rules for saving and recalling structured memory. It ensures memory quality, consistency, and retrieval accuracy regardless of which client (Claude, Codex, human UI) is interacting with Vault.

---

## 1. Save Protocol

When any client wants to save memory, the protocol is:

### Step 1: Decide if saving is warranted
Save when the result contains:
- A significant decision
- A session conclusion with useful outcomes
- A summary of work done
- A plan for future work
- A bug finding or debugging outcome
- Implementation progress worth preserving
- A handoff for session continuity
- An artifact worth future reuse

Do NOT save:
- Trivial filler
- Duplicates of already-saved content
- Noise or random fragments
- Incomplete work with no reuse value (unless intentionally kept as a handoff)

### Step 2: Determine the project
Every memory item should belong to a project when possible.

### Step 3: Determine memory type
Use controlled values: `session`, `summary`, `decision`, `plan`, `artifact`, `handoff`, `reference`

### Step 4: Determine the subject
Choose a specific main subject. Good: "SwiftFlow login redirect bug". Bad: "bug".

### Step 5: Determine keywords
Usually 3-8 good search terms. Include file names, function names, and module names when relevant.

### Step 6: Determine tags
Prefer controlled classification labels. Optionally add semantic tags.

### Step 7: Write summary
A short, reusable summary (2-5 sentences). A future session should understand this without extra context.

### Step 8: Add next steps if relevant
This is the most valuable field for continuity.

### Step 9: Determine save path
Automatically generated by Vault based on project + memory type + title.

### Step 10: Save file + register metadata
The file is stored under the Vault root. The registry (SQLite) is updated.

### Step 11: Optional enrichment
AI can: clean summary, improve tags, classify uncertain fields, suggest promotion.

### Step 12: Write log entry
Every save is logged with timestamp, client, project, and latency.

## 2. Recall Protocol

When a fresh session starts or a client needs prior context:

### Step 1: Read the current task
Extract: project clues, subject clues, feature/bug names, uncertainty signals, continuity signals.

### Step 2: Decide whether Vault should be queried
Triggers: "continue", "last time", project-specific work, uncertainty about prior decisions, likely prior context.

### Step 3: Search candidates
Filter by: project, subject/keywords, tags, memory type, status, recency.

### Step 4: Rank candidates
Deterministic scoring first:
- Project exact match: +50
- Subject exact match: +40
- Subject partial match: +20
- Keyword overlap: +10 per keyword
- Tag overlap: +8 per tag
- Promoted/canonical: +35
- Memory type priority: +5 to +20
- Priority level boost: +10 to +30
- Recency (7 days): +15
- Recency (30 days): +8
- Archived penalty: -20
- High access count (>5): +5

### Step 5: Return a memory pack
Do not return everything. Return:
- Top summaries/sessions
- Top decisions
- Top plans
- Optional artifacts/references

### Step 6: Let the client use it
The client integrates recalled memory naturally.

### Step 7: Log the recall
Store: who queried, what was returned, latency, candidate count.

## 2A. Async Task Protocol

Vault also supports queued asynchronous work through the task system.

Use this path when:
- the result should be produced later and stored in Vault
- the work is long-running or better handled as a background job
- another operator or later session may poll the result from the queue

### Step 1: Decide whether the work should be queued
Prefer queued tasks for summaries, delegated research, organization, or background artifact generation.
Prefer immediate chat responses when the result is needed right now in the same conversation.

### Step 2: Check executor availability
Before relying on queued work, inspect executor state with `vault_get_task_executor_status`.

### Step 3: Start an executor if needed
If the current MCP process is expected to execute the queue and its executor is stopped, start it with `vault_start_task_executor`.

### Step 4: Create the task
Use:
- `vault_create_task` for general delegated work
- `vault_request_summary` for summary generation over existing memory items

### Step 5: Poll and inspect
Use:
- `vault_get_task` for one task
- `vault_list_tasks` for queue browsing
- `vault_get_task_queue_stats` for queue pressure

### Step 6: Stop execution when appropriate
Use `vault_stop_task_executor` only when the current MCP process should stop consuming the queue.

### Important rule
Task creation is not execution. A task stays `pending` until some executor process is running. Vault now supports executor control from MCP as well as from the desktop app.

## 3. Subject vs Keywords vs Tags

| Field | Purpose | Cardinality | Example |
|---|---|---|---|
| subject | The one main topic name | 1 per item | "SwiftFlow login redirect bug" |
| keywords | Short search-friendly terms | 3-8 per item | ["login", "redirect", "auth"] |
| tags | Classification labels | 2-6 per item | ["bug", "backend", "auth"] |

These must not be confused. The distinction is central to retrieval quality.

---

## 4. Memory Lifecycle

```
Raw Memory → Structured Memory → Promoted Memory → Canonical Memory
```

- **Raw**: Immediate session outputs, quick saves
- **Structured**: Clean summaries, decisions, plans with proper metadata
- **Promoted**: Important items flagged for recall priority
- **Canonical**: Long-term stable project truths (architecture decisions, conventions)

---

## 5. Controlled Values

### Memory Types
`session`, `summary`, `decision`, `plan`, `artifact`, `handoff`, `reference`

### Routine Types
`debugging`, `planning`, `implementation`, `review`, `testing`, `brainstorming`, `refactor`, `deployment`

### Status Values
`active`, `resolved`, `draft`, `archived`, `promoted`

### Priority Values
`low`, `normal`, `high`, `critical`, `canonical`

### Source Apps
`claude`, `codex`, `openclaw`, `manual`, `other`

### Link Types
`related`, `informs`, `led_to`, `derived_from`, `supersedes`
